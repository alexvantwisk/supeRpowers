## How NAMESPACE Is Generated

Never edit `NAMESPACE` by hand. It is regenerated from roxygen2 comments
by `devtools::document()`. If you see `NAMESPACE` changes after editing
R files, it's because roxygen2 produced a different output — commit both
files together.

The first line of `NAMESPACE` should be:

```
# Generated by roxygen2: do not edit by hand
```

If that line is missing, run `devtools::document()` — it takes over the
file from then on.

## Import Strategies Compared

### Strategy 1: `pkg::fun()` (Default)

Use explicit `::` notation in your R code. No roxygen import needed.

```r
#' @export
summarise_data <- function(data, col) {
  data |>
    dplyr::group_by({{ col }}) |>
    dplyr::summarise(n = dplyr::n())
}
```

**Use when:** Default choice. Clear provenance, no NAMESPACE clutter,
easiest to audit dependencies.

**Cost:** Small lookup overhead per call (~100 ns). Negligible outside
hot loops.

### Strategy 2: `@importFrom pkg fun`

Import specific functions into your package namespace.

```r
#' @importFrom dplyr group_by summarise n
#' @export
summarise_data <- function(data, col) {
  data |> group_by({{ col }}) |> summarise(n = n())
}
```

**Use when:**
- Hot loops (> 10k calls) where `::` overhead matters
- Function called > 10 times in the same file
- Operators and infix functions (`|>`, `.data`, `:=`)
- Standard generics you override (`print`, `format`, `==`)

**Cost:** Names occupy your package namespace — collision risk if two
deps export the same name.

### Strategy 3: `@import pkg` (Avoid)

Imports everything from a package.

```r
#' @import dplyr        # Bad: pulls in all of dplyr
```

**Only use when:** You genuinely use > 20 functions from one package.
Almost never correct.

**Why it's bad:** Silent conflicts with other imports. `filter()` from
dplyr vs stats, `lag()` from dplyr vs stats, etc. Hard to audit.

## Centralising Imports

Consolidate common `@importFrom` statements in `R/mypkg-package.R` rather
than scattering them across files. Create it with `usethis::use_package_doc()`.

```r
#' @keywords internal
"_PACKAGE"

## usethis namespace: start
#' @importFrom rlang .data :=
#' @importFrom lifecycle deprecated
## usethis namespace: end
NULL
```

Benefits:
- Single source of truth for cross-cutting imports
- Easier review of what's imported
- `usethis::use_import_from()` adds here by default

## The `.data` Pronoun

When using dplyr/tidyr NSE with string column names, use `.data[[var]]`:

```r
#' @importFrom rlang .data
filter_by <- function(data, col_name, value) {
  data |> dplyr::filter(.data[[col_name]] == value)
}
```

Without the import, `R CMD check` reports:

```
no visible binding for global variable '.data'
```

Quick fix:

```r
usethis::use_import_from("rlang", ".data")
devtools::document()
```

## Tidy Eval with `{{ }}` (Curly-Curly)

`{{ }}` requires no imports — it's parsed by dplyr/rlang at runtime. You
only need to document the argument as data-masked:

```r
#' @param col <[`data-masking`][rlang::args_data_masking]> Column to filter.
filter_top <- function(data, col, n = 10) {
  data |>
    dplyr::arrange(dplyr::desc({{ col }})) |>
    dplyr::slice_head(n = n)
}
```

## Walrus Operator `:=`

Required when programmatically naming columns with `{{ }}` or `:=`.

```r
#' @importFrom rlang :=
rename_first <- function(data, new_name) {
  old_name <- names(data)[1]
  data |> dplyr::rename({{ new_name }} := !!old_name)
}
```

## Re-exporting Functions

Let your users call functions from a dependency through your package.

### Re-exporting a single function

```r
#' @importFrom tibble tibble
#' @export
tibble::tibble
```

Produces the NAMESPACE lines:

```
export(tibble)
importFrom(tibble,tibble)
```

### The base pipe

`usethis::use_pipe(type = "base")` sets `Depends: R (>= 4.1.0)` so the
base pipe `|>` is always available. No re-export file is created — it's
a language-level operator.

Older packages re-exported magrittr's pipe via a `utils-pipe.R` file
generated by `usethis::use_pipe(type = "magrittr")`. New packages should
require R >= 4.1.0 and use the base pipe exclusively.

## Function Name Conflicts

### Problem: My export `filter()` clashes with `dplyr::filter()`

You cannot force users to choose, but you can minimise friction:

**Option 1: Rename your function.** Usually the pragmatic choice.

```r
#' @export
filter_rows <- function(data, predicate) ...
```

**Option 2: Document the conflict prominently.**

```r
#' @section Note on masking:
#' This package's `filter()` masks `dplyr::filter()` when attached.
#' Call `mypkg::filter()` or `dplyr::filter()` explicitly to
#' disambiguate.
```

**Option 3: Use `conflicted` to force explicit choice.**

Add to README and vignettes:

```r
library(mypkg)
library(conflicted)
conflicts_prefer(mypkg::filter)
```

**Option 4: `.conflicts.OK`** (advanced, rarely appropriate)

In `R/zzz.R`, suppress the conflict message on attach:

```r
.onAttach <- function(libname, pkgname) {
  # Only for extreme cases
  packageStartupMessage(
    "mypkg masks dplyr::filter(). Use dplyr::filter() for the dplyr version."
  )
}
```

### Preventing internal conflicts

Inside your package code, always use `::` or `@importFrom` — never rely
on the search path. Your functions must work identically regardless of
what the user has attached.

## Internal Functions

Don't export helpers. Omit `@export`, keep them in `R/utils.R` or similar.

```r
# Not exported — no @export tag
validate_inputs <- function(x) {
  if (!is.numeric(x)) {
    cli::cli_abort("{.var x} must be numeric.")
  }
  invisible(x)
}
```

If you want the function visible in pkgdown reference index but hidden
from autocompletion:

```r
#' @keywords internal
#' @export
debug_state <- function() ...
```

### Accessing internals of other packages

**Don't use `:::`** in submitted code — CRAN rejects packages that
reach into another package's internals.

Alternatives:
1. Open an issue asking for the function to be exported.
2. Copy the function into your package (with attribution) if license allows.
3. Re-implement the logic.
4. Use a stable public API.

## S3 Method Registration

`@export` on a method registers it for dispatch.

```r
#' @export
print.temperature <- function(x, ...) { ... }
```

NAMESPACE gets:

```
S3method(print, temperature)
```

### S3 method for a generic in another package

Still just `@export` — roxygen2 detects the dotted name.

```r
#' @importFrom tibble tbl_sum
#' @export
tbl_sum.temperature <- function(x, ...) {
  c("A temperature" = pillar::dim_desc(x))
}
```

---

## S4 Method Registration and `@include`

S4 requires classes and generics to be defined before methods. Use
`@include` to control load order — roxygen2 populates the `Collate:`
field in DESCRIPTION.

```r
# R/classes.R
setClass("GenomicRange", ...)

# R/methods.R
#' @include classes.R
setMethod("width", "GenomicRange", function(x) x@end - x@start + 1L)
```

Check generated `Collate:` matches what you expect after
`devtools::document()`.

## Auditing Your NAMESPACE

After `devtools::document()`, scan for patterns that suggest cleanup:

```r
usethis::use_tidy_dependencies()   # Tidyverse-style cleanup
pkgdepends::pkg_name_check()       # Validate package name
```

Useful manual checks:

```bash
grep '^import(' NAMESPACE         # Catch accidental whole-package imports
grep '^importFrom' NAMESPACE | sort | uniq -c | sort -n
grep '^export' NAMESPACE | wc -l  # Count your public API surface
```

## Summary Decision Table

| Scenario | Strategy |
|----------|----------|
| Used once or twice in the package | `pkg::fun()` |
| Used in a hot loop | `@importFrom pkg fun` |
| Operator (`|>`, `:=`, `.data`) | `@importFrom` in pkg-package.R |
| Standard generic override (`print`, `==`) | `@importFrom` |
| Deep integration with another package | Consider `Depends`, but usually still `Imports` |
| Clashes with user's other deps | Rename or recommend `conflicted` |
| Need another package's internal | Request export upstream or re-implement |

# supeRpowers Plugin — Consulting Audit Report

**Date:** 2026-04-16
**Scope:** 22 skills, 5 agents, 1 rule, 1 hook, Python test suite
**Method:** Direct read of all plugin artefacts; parallel research against Anthropic's Skills spec (Oct 2025), obra/superpowers v5.0.7, wshobson's evaluation rubric, and current R power-user expectations (ellmer, btw, mcptools, air, R 4.5, duckplyr)
**Deliverable:** Three gaps, existing strengths to amplify, three adversarial removal candidates

---

## 1. Three Gaps That Will Materially Improve the Plugin

### Gap A — No skill for the R + AI stack itself (ellmer / chattr / gander / ragnar / vitals)

This is the highest-leverage miss and, given the plugin's positioning, the most embarrassing.

`ellmer` is the Posit/tidyverse LLM client that "defaults to Claude 4.0 Sonnet, which has been found to be particularly good at writing R code" (ellmer.tidyverse.org). `chattr` and `gander` are the IDE-side streaming assistants. `ragnar` is the canonical R RAG package; `vitals` is the 2025 LLM-eval package.

An R plugin shipped in 2026 that teaches Claude everything about R *except* how R talks to Claude is strategically incoherent.

**Action:** Add `r-ai-integration` covering ellmer chat/tool-calling/structured-output, ragnar ingestion, and vitals evaluation — cross-linked from `r-mcp-setup`.

### Gap B — No database skill (DBI / dbplyr / duckdb / duckplyr)

Databases are the single largest non-tidyverse R workload, and the plugin has nothing. `duckplyr` (Nov 2025) specifically is "fully native dplyr directly coupled to DuckDB's query engine" and is changing how R people query data at scale.

**Action:** Add `r-databases` covering DBI connection patterns, dbplyr lazy evaluation, `collect()` timing, SQL translation gotchas, and duckdb/duckplyr for local analytics. This workload is currently being forced into `r-data-analysis` where it doesn't belong.

### Gap C — No web/API/deployment skill (httr2 / plumber / vetiver)

`httr2` is the modern successor to httr and one of the most-used packages the plugin doesn't cover. `plumber` exposes R as an API; `vetiver` is Posit's MLOps layer pairing directly with `r-tidymodels`.

The plugin teaches users to build models but not to deploy them, and teaches data analysis but not to fetch data from REST APIs.

**Action:** One `r-apis-deployment` skill closes both loops.

## 2. Already-Strong Parts Worth Doubling Down On

Power-user skill quality clusters around the obra/superpowers playbook: verb-first naming, trigger-first "Use when…" descriptions, progressive disclosure via `references/`, and concrete worked examples. Focus investment here:

### 2a. `r-mcp-setup` — strategically the most valuable skill; currently only tactical

It already documents the btw + mcptools wiring, which Simon Couch's widely-cited post (*simonpcouch.com/blog/2025-07-17-claude-code-2*) identifies as *the* Claude-Code-plus-R killer integration. Upgrade priorities:

- Add `paths:` frontmatter gating (`*.R`, `*.Rmd`, `*.qmd`, `DESCRIPTION`, `NAMESPACE`) so MCP guidance isn't loaded for non-R sessions.
- Rewrite the description to be trigger-first ("Use when the user is slow-iterating in an R session, needs live data-frame inspection, or asks 'can you just run this?'") rather than feature-first ("Use when the user wants to set up MCP tools").
- Add a runnable bootstrap script (`scripts/bootstrap_mcp.R`) so setup is one command, not a five-step walkthrough.

### 2b. `r-data-analysis`, `r-tdd`, `r-code-reviewer` agent — already at or near gold standard

These three are the plugin's showcase artefacts.

- `r-data-analysis` has ~65% concrete code, a real gotchas table, both happy-path and edge-case examples (the join-fan-out example at `skills/r-data-analysis/SKILL.md:172-189` is genuinely instructive).
- `r-tdd` matches obra's red-green-refactor doctrine closely.
- `r-code-reviewer` has a proper I/O contract, severity rubric, and explicit escalation table to `r-statistician` / `r-dependency-manager`.

**Remaining lift:** add vdiffr snapshot tests for `r-visualization`; add an explicit "Iron law: never edit implementation before the test fails" callout to `r-tdd` borrowed verbatim from obra; extend `r-code-reviewer`'s checks to flag `air`-style infractions, not just `styler`-era ones.

### 2c. `rules/r-conventions.md` — tight, but one convention is already stale

159 lines, imperative, well-structured. **But it still names `styler` as the formatter.**

`air` (Posit, Feb 2025; v0.7.0 Jun 2025; bundled in Positron) has replaced `styler` as the default tidyverse-style formatter in 2026.

**Action:** Update the Style section to reference `air format .` and `Posit.air-vscode`; mention R 4.5's native `grepv()` and parallel `install.packages()`; point at `pak` as the renv install backend (`renv.config.pak.enabled = TRUE`). Small edits, outsized credibility gains with R power users.

### 2d. Test infrastructure (`tests/routing_matrix.json`, `test_conventions.py`, `test_structural.py`)

The adversarial routing matrix — 42 prompts, 9 categories including survival-analysis and regression-vs-ML ambiguity — is unusually disciplined for a Claude Code plugin, ahead of most community plugins.

**Action:** Wire it into CI and publish the pass rate in README as a quality signal (wshobson-style Platinum / Gold / Silver badge).

## 3. Adversarial Review — Three Things That Should Be Removed

### Remove 1 — The entire `r-cmd-*` cluster (5 skills)

**Files:** `r-cmd-tdd-cycle`, `r-cmd-debug`, `r-cmd-analysis`, `r-cmd-shiny-app`, `r-cmd-pkg-release`

The design spec at `docs/superpowers/specs/2026-04-01-commands-layer-design.md` explicitly justifies these with:

> *"Since Claude Code plugin.json does not support a separate `commands` key, workflow commands are implemented as skills with a distinct naming convention (`r-cmd-*`)."*

**This premise is wrong.** Claude Code plugins support both a `commands/` directory (flat `.md` files — legacy) and `skills/<name>/SKILL.md` with `user-invocable: true` for native slash-command registration (Claude Code Plugins Reference, *code.claude.com/docs/en/plugins-reference*).

The `r-cmd-*` files advertise "Invoke as /r-cmd-X" in their descriptions, but they are not registered as commands anywhere — the slash-command invocation they promise does not exist as described.

**Adversarially:** these five files burn frontmatter budget (Anthropic's 1%-of-context cap), duplicate content already present in their underlying knowledge skills (e.g. `r-cmd-tdd-cycle`'s Step 2 RED / Step 3 GREEN reiterate `skills/r-tdd/SKILL.md:34-38`), and introduce a fictitious skill taxonomy ("knowledge skills vs workflow skills") that has no parallel in obra/superpowers or Anthropic's own `skills` repo.

**The correct move:** either (a) convert them to real slash commands under `commands/` or as `user-invocable: true` skills, or (b) delete them and trust the underlying skills to orchestrate via the existing agent-dispatch lines. Keeping them as-is is the worst of both worlds.

### Remove 2 — `skill-auditor`

A sophisticated 38-check meta-skill with Python scoring scripts, parallel subagent dispatch for batch auditing, and a whole `references/remediation-guide.md`. It is genuinely well-built.

But **it is out of scope for an R programming plugin.** supeRpowers' users are R programmers solving R problems; they are not authoring Claude Code skills. Shipping `skill-auditor` in this plugin means every R user's skill-listing budget carries a meta-tool they will never invoke.

**Adversarially:** the `disable-model-invocation: true` flag in its frontmatter is an implicit admission that it doesn't belong in auto-activation territory. This belongs in its own `claude-skills-toolkit` plugin, or relocated to `scripts/` as a maintainer-only build tool (outside `plugin.json`'s skill glob).

### Remove 3 — `r-package-skill-generator`

Another meta-tool. Python scaffolding (`setup_workspace.py`, `scan_r_package.py`), parallel exploration agents, 256-line orchestration SKILL.md. It generates Claude skills from arbitrary GitHub R packages.

Impressive; useful to the *maintainer* of this plugin; useless to the R data scientist who installed supeRpowers to clean a CSV.

**Adversarially:** it is a plugin-authoring tool masquerading as an end-user R skill. Its own description even warns *"Do NOT use for general R package development — use r-package-dev instead"* — which is the confession. The Python script inventory in `skill-auditor/scripts/` + `r-package-skill-generator/scripts/` totals tens of KB of non-R tooling inside an R plugin.

**Action:** Relocate both `skill-auditor` and `r-package-skill-generator` to a separate `plugin-authoring-toolkit` marketplace entry. The supeRpowers plugin gets tighter, more focused, and stops signalling "this is a plugin about building plugins."

## 4. Professional Assessment

Staking my reputation:

**The plugin's core — 15 domain skills + 5 agents + conventions rule + MCP skill + session-start hook — is above-median quality for the Claude Code ecosystem in 2026, comparable to wshobson Gold tier after the §2 fixes land.**

What's dragging it down is accretion:

- The `r-cmd-*` experiment (5 skills built on a false premise about commands support)
- Two sophisticated meta-tools that belong in a different plugin
- A stale formatter reference (`styler` → `air`)
- An R + AI gap that undermines the product positioning

Remove the §3 items, add the §1 skills, ship the §2 polish, and this becomes the reference R plugin for Claude Code.
